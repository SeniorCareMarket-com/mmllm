# FIM run — JSON 10k steps

## Setup

- date (UTC): 2026-05-10 ~21:44 → ~22:12 (train 21:12 wall)
- language: json
- source: **glaive-function-calling-v2** (Salesforce/xlam-function-calling-60k
  is HF-gated and prepare-hf-dataset failed with `DatasetNotFoundError`;
  fell back to the next-largest JSON tool-call corpus). 200 MB raw → 20 000
  `<|sys|>...<|end|>` transcripts unpacked into per-doc `.json` files
  (filter 100 < len < 4096).
- splitter: `json-value-boundary`
- fim_ratio: 0.7  /  psm_ratio: 0.5
- steps: 10 000  /  batch: 4  /  seq_len: 128
- tokens: 5 120 000
- corpus: 13 930 FIM + 6 070 raw-causal docs, train=30.8 MB, val=2.8 MB
- bank: cpu-tiny mmap (`/tmp/mmllm-cpu/fim-json-bank.{0..3}.bin`, ~30 MB each)
- pipeline fixes pulled in (separate branch `claude/fim-eval-agent-cfg-fix`):
  1. eval-agent autodetects cpu-tiny config from ckpt's tok_emb shape
     (otherwise shape mismatch at load)
  2. eval-agent caps `max_prompt_bytes` at `max-pos - gen-len - 8`
     (otherwise RoPE-cache overflow at first attention)
  3. eval-agent logs sample count + optional per-sample
     `MMLLM_AGENT_EVAL_VERBOSE=true` prediction dump

## Results (raw)

### Train trajectory

| | value |
|---|---|
| final train loss (step 10 000) | 1.100 |
| final val_bpc (step 10 000) | **1.5675** (ppl 2.96) |
| val_bpc with bank zeroed (ablation) | 3.6443 (Δ +2.08 — bank carries signal) |
| tps (steady state) | ~5 000 tok/sec |

| energy estimate | 0.0234 kWh / 12.8 gCO₂eq |

Loss dropped from ~5.5 (random) → ~1.1 by step 10k, hitting bpc ≤ 2 by
step 5 000. FIM loss mask was confirmed firing (`train-fim:
FIM_LOSS_MASK_MIDDLE_ONLY=true forced`).

The pattern is the predicted one: **json (the training language) has the
strongest FIM-bpc of any row** (3.85 vs 6.27–6.32 for unseen langs), and
clojure / python sit at near-random byte entropy (6 ≈ 8-bit uniform − a
bit). But OVERALL bpc 4.80 is well above the runbook's "converged"
threshold of 2.0, and FIM-exact is 0/0/0/0 — no full-middle was
reproduced verbatim.

### Agent eval (glaive, 10 samples reaching the model)

```
eval-agent[glaive]: ckpt=10000 test=/tmp/mmllm-cpu/sources/glaive.test.bin n=100 gen=128
iter_eval_samples returned 10 samples (max_prompt_bytes=376)
→ format=0.000  name=0.000  args=0.000  exact=0.000  (7.2s)
```

**Two caveats on the agent eval setup itself**, both flagged here so we
don't mistake them for the hypothesis verdict:

1. cpu-tiny's `max-pos=512` capped usable prompt bytes at 376
   (`max-pos − gen-len − 8`). Of the 1 994 glaive transcripts in
   `glaive.test.bin`, only **10** (0.5 %) had prompts short enough to
   feed the model. Median prompt length is 1 487 B — the system message
   + tool definitions overwhelm the window.
2. Of all 1 994 glaive test samples, **zero have `tool_calls` in their
   gold assistant turn**. They're the "I'm sorry, I can't order pizza"
   refusal class. So `tool_name_match` / `tool_args_match` / `exact_match`
   are structurally pinned at 0 regardless of model quality.
   `format_validity` is the only metric here that can still move (it
   checks if the model spontaneously emits a parseable
   `{"tool_calls":[…]}` envelope).

Representative decoded outputs at step 10 000 (greedy decode, gen-len 128):

```
[sample 0/5]
  PROMPT (last 100B): '...{"name": "get_random_joke", ...}\n<|end|>\n<|user|>\nCan you order a pizza for me?\n<|end|>\n<|asst|>\n'
  PRED:  '<|\n{de cuthtampacinopfunon amano a", fithemasetinti", tatacs adotata
          nomomoxieno_poretenoc widodairetitanocaca o ieta taiayonoco'
  GOLD:  "I'm sorry, but as an AI, I don't have the capability to perform external
          tasks such as ordering a pizza. ..."

[sample 4/5]
  PRED:  'etha tamp astemendowsetowicholetethetwot <|>\n<|>: tirometa amppondoratetotat
          hematithecacano iththethoisthapa", wino_no thecanome'
```

The model has learned that `<|`, `{`, `"`, `,`, `:` are frequent — there
are clearly mark-style and JSON-structural priors firing — but it cannot
yet assemble a syntactically valid envelope, never mind one matching the
schema in the system message.

## Decision

**Decision matrix row that applies:**

> `format_validity = 0.0` AND `FIM-bpc OVERALL ≥ 4.0` (after ≤ 30k steps) →
> **Training under-converged.** Recommendation: run a 50 k extension before
> claiming refutation.

The hypothesis (bidirectional structural conditioning enables valid JSON
envelope synthesis) is **neither confirmed nor refuted** by this run.
The model didn't get enough updates to produce coherent JSON — and the
"can it produce coherent JSON at all" question has to be answered
positively before the headline "does FIM unlock structural fill" can
even be tested. The signal that matters is faintly visible (json-row
bpc is meaningfully lower than the other language rows), but the
overall convergence floor is too high.

A 50 k or 100 k continuation from this ckpt is the cheapest next move.
Bank ablation (val_bpc 1.57 → 3.64) confirms the bank is doing real
work, so the model architecture is engaging — it's purely a step-count
problem.

## Notes / surprises

- **xlam was HF-gated.** `prepare-hf-dataset xlam` exits with
  `DatasetNotFoundError: ... is a gated dataset on the Hub. You must
  be authenticated to access it.` Falling back to glaive-funcall
  produced 200 MB → 20 k training docs at the right JSON shape, but
  glaive's test split is dominated by refusal-class transcripts and
  long system messages, which is a poorer fit for the cpu-tiny eval
  window than xlam would have been. Future runs should either get a
  HF token wired in or unpack a positive-class slice of glaive/hermes
  manually before generating `.test.bin`.
- **Two real pipeline bugs surfaced while wiring the eval-agent
  step:** (a) the CLI handler picked `default-config` instead of
  `default-config-cpu-tiny`, causing a `.copy_` shape mismatch on the
  trained dense.pt; (b) `iter_eval_samples` defaulted to
  `max_prompt_bytes=768` while cpu-tiny's `max-pos` is 512, so the
  first attention forward threw `start (0) + length (518) exceeds
  dimension size (512)`. Both fixed and committed on
  `claude/fim-eval-agent-cfg-fix` (cherry-picked into this run
  branch). A third commit adds an `MMLLM_AGENT_EVAL_VERBOSE` flag
  that prints predictions per sample — the diagnostic that revealed
  the gibberish-output pattern instead of just a metric of zero.
- **The bank ablation (control bpc 1.57 vs ablated 3.64) is the
  strongest end-of-run signal:** the FIM bank is carrying ~2 bpc of
  learned structural information. That's the architecture working;
  the model just hasn't trained long enough for that information to
  surface as decodable JSON envelopes.
- **Energy:** 0.0234 kWh / 12.8 gCO₂eq for the 10 k run. A 50 k
  extension is ~5× that.
